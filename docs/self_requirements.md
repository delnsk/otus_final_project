# Дополнительные требования к проекту

Документ дополняет [исходное задание](source_task.md) собственными требованиями к реализации.

## 1. Требования по логированию

### 1.1. Логирование MCP

- **Каждый вызов MCP-инструмента** должен логироваться вместе с **параметрами** этого инструмента (имя инструмента, входные аргументы, результат или ошибка).
- **Запрос списка инструментов** MCP (discovery / `list_tools` и аналоги) также должен логироваться.
- **При запуске MCP-сервера** в лог записываются **параметры запуска**: конфигурация, порт, пути, переменные окружения (без секретов), режим работы и прочие значимые настройки, с которыми поднят сервер.

### 1.2. Логирование многошаговых пайплайнов

Для запросов с несколькими преобразованиями (в первую очередь — Corrective RAG в `ask_question`) **каждый шаг** выполнения должен логироваться **с промежуточным результатом**.

Пример пайплайна (из задания):

```
User Query → Rewrite Query → Retrieve (hybrid: BM25 + vector → RRF)
  → Grade Chunks (relevant? yes/no)
    → enough relevant → Generate Answer → Return answer + sources
    → too few relevant → Broaden Query → Retrieve (max 2 loops)
```

Для каждого шага в лог попадают как минимум:

| Шаг | Что логировать |
| --- | --- |
| User Query | Исходный вопрос пользователя |
| Rewrite Query | Переформулированный запрос |
| Retrieve | Запрос, top-k, список чанков (id/источник/скор), сводка по BM25 и vector до RRF |
| Grade Chunks | Оценка каждого чанка (relevant: yes/no), итоговое число релевантных |
| Broaden Query | Расширенный запрос и номер итерации цикла |
| Generate Answer | Сгенерированный ответ (или его сокращённая версия) и список источников |

Аналогичный принцип применяется к другим многошаговым операциям (например, `index_folder`: сканирование → загрузка → чанкинг → эмбеддинги → запись в ChromaDB).

### 1.3. Ротация лог-файлов на бэкенде

- Логи на бэкенде пишутся в файлы с **ротацией по размеру: 1 МБ на файл**.
- При достижении лимита создаётся новый файл; старые сохраняются для последующего анализа.

### 1.4. Просмотр логов в реальном времени

- Поднимается **отдельный HTTP-порт** только для просмотра логов.
- При открытии в браузере отображаются **последние 200 строк** из логов.
- Обновление — **в реальном времени** (tail / WebSocket или SSE), без перезагрузки страницы.

## 2. Требования к MCP

### 2.1. Асинхронный протокол

- MCP-сервер должен работать по **асинхронному протоколу** (async I/O, неблокирующая обработка запросов).
- Вызовы инструментов и транспорт не должны блокировать event loop при ожидании LLM, индексации или обращений к хранилищу.

## 3. Требования к архитектуре

### 3.1. SOLID

При написании кода необходимо руководствоваться принципами **SOLID**:

- **S** — единственная ответственность: отдельные модули для индексации, retrieval, графа, MCP-обёртки, логирования.
- **O** — открытость/закрытость: расширение (новые форматы, retriever'ы) без правки ядра.
- **L** — подстановка: абстракции (LLM, vector store, logger) заменяемы в тестах и конфиге.
- **I** — разделение интерфейсов: узкие контракты вместо «божественных» классов.
- **D** — инверсия зависимостей: зависимость от абстракций, внедрение через конструктор/фабрику/конфиг.

### 3.2. Покрытие тестами

- Всё, что **можно** покрыть **юнит-** или **интеграционными** тестами, должно быть покрыто.
- Минимум из [исходного задания](source_task.md) (≥ 10 тестов) сохраняется; дополнительно — тесты на логирование, ротацию файлов, шаги графа с mock LLM, MCP-инструменты (e2e).
