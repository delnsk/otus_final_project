# Отчёт о разработке

## История

Проект реализован поэтапно по плану [`docs/implement_plan.md`](docs/implement_plan.md) — 15 фаз от каркаса Python-проекта до CI и документации для сдачи.

| Этап | Что сделано |
| --- | --- |
| 0–1 | Каркас (`pyproject.toml`), `Settings`, DI-контейнер |
| 2–3 | Доменные модели, порты, RRF, логирование с ротацией 1 МБ |
| 4–6 | Загрузчики 7 форматов, чанкеры, ChromaDB, гибридный retrieval |
| 7–8 | Сервисы индексации/поиска, LangGraph Corrective RAG |
| 9–10 | FastMCP (4 инструмента, middleware), Log Viewer (SSE) |
| 11 | Docker Compose: Ollama + model-init + rag-mcp-server |
| 12 | Демо-корпус `sample_docs/` (~1.9 МБ), проверочные факты |
| 13 | 76 тестов: unit, integration, e2e MCP |
| 14 | GitHub Actions: ruff + pytest + docker build |
| 15 | README, REPORT, ARCHITECTURE (корень + docs/) |

Архитектурный стиль — гексагональная компоновка (Ports & Adapters): MCP и Log Viewer на входе, сервисы приложения, домен с LangGraph-графом, инфраструктурные адаптеры (Ollama, ChromaDB, BM25) на выходе.

## Показательный промпт (удачный)

```
Выполни шаг за шагом пункты 1-13 из docs/implement_plan.md.
После каждого шага выполни обязательную проверку из плана.
```

**Почему сработал:** план уже содержит критерии приёмки (pytest, integration-тесты, docker build). Агент не «додумывал» архитектуру с нуля, а следовал готовому `ARCHITECTURE.md` и чеклистам — меньше расхождений с требованиями сдачи.

**Результат:** 54→76 тестов green, все фазы 0–15 закрыты за несколько сессий. После каждой фазы запускались проверки; баги исправлялись до перехода дальше.

**Урок:** явное «проверяй после каждого шага» + ссылка на документ с критериями эффективнее общего «сделай RAG MCP-сервер».

## Ключевые проблемы и решения

### Инфраструктура и окружение

1. **Долгая первая сборка Docker** — при первом `docker compose up` сервис `model-init` скачивает LLM и модель эмбеддингов (`phi3:mini`, `nomic-embed-text`); сборка образа + pull моделей занимает 10–20+ минут. **Решение:** ожидаемое поведение; в README явно указано, что первый запуск долгий; повторные запуски используют volume `ollama_models`.

2. **`docker compose up` падает, если Docker не запущен** — на Windows ошибка «Cannot connect to the Docker daemon». **Решение:** перед запуском нужен работающий Docker Desktop; добавлено в README в раздел «Быстрый старт».

8. **Размазанная конфигурация без единой точки настройки** — параметры дублировались в `Dockerfile`, `docker-compose.yml` и `config.py`; не было `.env.example`. **Решение:** единый шаблон [`.env.example`](.env.example) → `.env`; compose подставляет `${VAR:-default}`; `config.py` читает тот же `.env` через pydantic-settings; из `Dockerfile` убраны runtime-ENV с моделями. Документировано в README и ARCHITECTURE.

### Качество RAG и производительность

3. **`ask_question` на слабой машине (Intel Core i5-1135G7, 16 ГБ RAM) часто падает по таймауту** — пайплайн Corrective RAG делает до 4+ вызовов LLM (rewrite, grade по каждому чанку, broaden, generate); на CPU-only Ollama укладывается дольше httpx-timeout (120 с). **Частичное решение:** timeout 120 с в `OllamaLLM` и эмбеддингах; для сдачи рекомендуется индексировать подмножество (`sample_docs/book` вместо всего корпуса) или дождаться завершения. Полностью на CPU без GPU остаётся ограничением железа.

4. **`ask_question` с Phi-3-mini даёт неточные ответы и галлюцинации** — малая модель (3.8B) слабо справляется с grade-шагом и генерацией по длинному контексту. **Решение:** в `.env.example` и README указана альтернатива `LLM_MODEL=qwen2.5:3b`; переключение одной строкой без правки кода. Phi-3 оставлен дефолтом для быстрого pull; для проверочных фактов лучше Qwen.

### Согласованность индекса

5. **После перезапуска часть индекса «терялась»** — симптом: чанки есть в ChromaDB (vector search находит), но `index_status` показывает 0 файлов, BM25 не находит те же документы. Причины: BM25 хранился только в памяти; `file_count` и `last_indexed_at` не пересчитывались из персистентных метаданных Chroma. **Решение:** при старте `Container` гидратирует BM25 из всех чанков Chroma (`hybrid_retriever.hydrate`); после каждого `index_folder` — полная пересборка BM25 из store; `get_stats()` считает `file_count` как distinct `source` по метаданным коллекции, `last_indexed_at` пишется в metadata коллекции. Покрыто integration-тестами `test_index_restart.py`.

### MCP и документация

6. **У MCP-инструментов не было описаний параметров** — хост-агент не понимал, что передавать в `path`, `glob`, `top_k`. **Решение:** параметры объявлены через `Annotated[..., "описание"]` в `mcp/tools.py`; e2e-тест `test_mcp_tool_parameter_descriptions` проверяет наличие `description` в JSON Schema.

7. **Агент создал фейковый REPORT.md без запроса** — ранний черновик содержал выдуманную историю и несуществующие проблемы. **Решение:** отчёт переписан вручную по фактическим сессиям из [`docs/prompts.md`](docs/prompts.md); раздел «Ключевые проблемы» пополняется только реальными инцидентами.

### Технические баги (исправлены в коде)

9. **pydantic-settings** — без `populate_by_name=True` нельзя передать `chroma_path` в конструктор `Settings`; тесты использовали дефолтный путь и влияли друг на друга. Решение: флаг в модели + изолированные temp-директории в тестах.

10. **YAML dates** — `yaml.safe_load` парсит даты как `datetime.date`; JSON-чанкер падал при `json.dumps`. Решение: `default=str` при сериализации.

11. **Логирование на Windows** — кастомные `LogRecord` не всегда попадали в файл. Решение: прямая запись строк в pipeline logger.

12. **Launcher pip/pytest на Windows** — `pip.exe` и `pytest.exe` указывали на удалённый каталог Python. Решение: везде `python -m pip` / `python -m pytest`.

13. **Демо-корпус** — сгенерированные тексты заменены на связные детективы и реальный open-source код `deepagents` для правдоподобных проверочных фактов.

14. **`ARCHITECTURE.md` не в корне проекта** (требование сдачи) — файл был только в `docs/`. Решение: добавлен [`ARCHITECTURE.md`](ARCHITECTURE.md) в корень с обзором и ссылкой на полную версию в `docs/`.

## Вопросы

- Оптимальный размер LLM для Corrective RAG на CPU-only (phi3:mini vs qwen2.5:3b) — trade-off скорость/качество grade-шага.
- Нужен ли настраиваемый `OLLAMA_TIMEOUT` в `.env` для слабых машин.
- Нужен ли отдельный healthcheck HTTP-endpoint для Docker вместо только Log Viewer.
- Стоит ли в CI добавлять job с реальным Ollama для smoke-теста `ask_question` (сейчас e2e с mock).
