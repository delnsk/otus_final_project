# Отчёт о разработке

## История

Проект реализован поэтапно по плану [`docs/implement_plan.md`](docs/implement_plan.md) — 15 фаз от каркаса Python-проекта до CI и документации для сдачи.

| Этап | Что сделано |
| --- | --- |
| 0–1 | Каркас (`pyproject.toml`), `Settings`, DI-контейнер |
| 2–3 | Доменные модели, порты, RRF, логирование с ротацией 1 МБ |
| 4–6 | Загрузчики 7 форматов, чанкеры, ChromaDB, гибридный retrieval |
| 7–8 | Сервисы индексации/поиска, LangGraph Corrective RAG |
| 9–10 | FastMCP (4 инструмента, middleware), Log Viewer (SSE) |
| 11 | Docker Compose: Ollama + model-init + rag-mcp-server |
| 12 | Демо-корпус `sample_docs/` (~1.9 МБ), проверочные факты |
| 13 | 58 тестов: unit, integration, e2e MCP |
| 14 | GitHub Actions: ruff + pytest + docker build |
| 15 | README, REPORT, актуализация ARCHITECTURE |

Архитектурный стиль — гексагональная компоновка (Ports & Adapters): MCP и Log Viewer на входе, сервисы приложения, домен с LangGraph-графом, инфраструктурные адаптеры (Ollama, ChromaDB, BM25) на выходе.

## Показательный промпт (удачный)

```
Выполни шаг за шагом пункты 1-13 из docs/implement_plan.md.
После каждого шага выполни обязательную проверку из плана.
```

**Почему сработал:** план уже содержит критерии приёмки (pytest, integration-тесты, docker build). Агент не «додумывал» архитектуру с нуля, а следовал готовому `ARCHITECTURE.md` и чеклистам — меньше расхождений с требованиями сдачи.

**Результат:** 54→58 тестов green, все фазы 0–13 закрыты за несколько сессий. После каждой фазы запускались проверки; баги (pydantic `populate_by_name`, сериализация дат в YAML-чанкере) исправлялись до перехода дальше.

**Урок:** явное «проверяй после каждого шага» + ссылка на документ с критериями эффективнее общего «сделай RAG MCP-сервер».

## Показательный промпт (неудачный)

```
@c:\Users\admin\.cursor\projects\...\terminals\7.txt:10-14
```

(Ссылка только на фрагмент терминала без формулировки задачи.)

**Почему не сработал с первого раза:** агент должен был угадать намерение по traceback. Потребовался уточняющий диалог.

**Исправление:** в follow-up явно описали проблему (`pip.exe` ссылается на удалённый Python) — решение: `python -m pip` вместо `pip`. Это отражено в README (раздел «Разработка»).

## Ключевые проблемы и решения

1. **pydantic-settings** — без `populate_by_name=True` нельзя передать `chroma_path` в конструктор `Settings`; тесты использовали дефолтный путь и влияли друг на друга. Решение: флаг в модели + изолированные temp-директории в тестах.

2. **YAML dates** — `yaml.safe_load` парсит даты как `datetime.date`; JSON-чанкер падал при `json.dumps`. Решение: кастомный `default=str` при сериализации.

3. **Логирование на Windows** — кастомные `LogRecord` не всегда попадали в файл. Решение: прямая запись JSON-строк в pipeline logger.

4. **Launcher pip/pytest на Windows** — `pip.exe` и `pytest.exe` указывали на удалённый каталог Python. Решение: везде `python -m pip` / `python -m pytest`.

5. **Демо-корпус** — сгенерированные тексты Zephyr заменены на связные детективы и реальный open-source код `deepagents` для правдоподобных проверочных фактов.
